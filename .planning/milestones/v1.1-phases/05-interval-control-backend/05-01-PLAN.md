<objective>
Remove the static `beat_schedule` dict from `celery_app.py` and replace it with DB-backed intervals that persist across Beat restarts. Seed default intervals and minimums into `system_config` on first boot. Bootstrap RedBeat entries from DB before Beat starts its tick loop.

Purpose: FREQ-03 requires that poll interval settings persist across Beat restarts. The current `beat_schedule` dict causes RedBeat's `setup_schedule()` to overwrite operator-configured intervals every time Beat restarts. This plan eliminates that coupling.

Output: Modified celery_app.py (no beat_schedule), modified seed.py (interval seeding), new beat_bootstrap.py, modified docker-compose.yml (beat command with bootstrap step).
</objective>

The `bootstrap_beat_schedule()` function:
1. Opens a `SyncSessionLocal` session
2. Reads all `SystemConfig` rows into a dict
3. For each worker in the map, reads `poll_interval_{worker}` from the dict
4. Creates a `RedBeatSchedulerEntry` with `name=beat_name`, `task=task_name`, `schedule=celery_schedule(float(interval_seconds))`, `app=celery_app`
5. Calls `entry.save()` — which uses `hset` for definition (overwrites interval) and `hsetnx` for meta (preserves `last_run_at`)
6. Logs each entry written

**1. Modify celery_app.py — remove the beat_schedule dict:**

Remove the entire `beat_schedule` key from the `celery_app.conf.update()` call. This includes the "Beat schedule" comment and all 6 schedule entries (lines 41-67 of the current file).

CRITICAL anti-pattern from RESEARCH.md Pitfall 1: If `beat_schedule` is left in place (even empty `{}`), RedBeat's `setup_schedule()` calls `update_from_dict()` which could interfere with bootstrap-written entries. However, per RESEARCH.md Open Question 1 and Anti-Pattern analysis in Pitfall 2: setting `beat_schedule={}` is actually safer than removing the key entirely because it ensures `app.redbeat_conf.schedule` returns `{}` consistently.

Decision: Set `beat_schedule={}` explicitly with a comment explaining why. This ensures:
- `update_from_dict({})` is a no-op (no entries to overwrite)
- `setup_schedule()` statics cleanup: `previous.difference({})` = `previous` — old statics get cleaned up
- Bootstrap entries were never in `statics_key` (they use `entry.save()` directly, not `update_from_dict`), so they survive cleanup

Add a comment block explaining the reasoning:
```python
# Beat schedule is intentionally empty — intervals are DB-backed (FREQ-03).
# beat_bootstrap.py writes RedBeat entries from system_config before Beat starts.
# DO NOT add entries here — they would overwrite DB-configured intervals on restart.
beat_schedule={},
```

Keep the existing `beat_scheduler`, `redbeat_redis_url`, `redbeat_lock_timeout` settings unchanged.
Keep the startup assertion for RedBeatScheduler unchanged.

**2. Modify docker-compose.yml — Beat container command:**

Change the `beat` service command from:
```
celery -A app.workers.celery_app beat --scheduler redbeat.RedBeatScheduler --loglevel=info
```
To:
```
bash -c "python -m app.workers.beat_bootstrap && celery -A app.workers.celery_app beat --scheduler redbeat.RedBeatScheduler --loglevel=info"
```

This runs the bootstrap script before Beat starts its tick loop. The `&&` ensures Beat only starts if the bootstrap succeeds. If the bootstrap fails (DB unreachable, etc.), the container will exit with an error rather than starting Beat with no schedule entries.

Also add `postgres` as a dependency for the `beat` service (currently only depends on `redis`). The bootstrap reads from `system_config` which is in PostgreSQL, so Beat needs Postgres to be healthy before starting:
```yaml
beat:
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
```

Also add `backend` as a dependency (condition: not health-checked, just started) so that `alembic upgrade head` and `seed.py` have run before beat_bootstrap tries to read system_config. Actually, the existing `backend` service runs `alembic upgrade head && python -m app.seed && uvicorn` — the beat service should wait for backend to be up to ensure migrations and seed have run. Use `depends_on: backend` without a health condition (service_started is the default), which ensures the backend container has at least started (running alembic/seed).

Wait — that creates a circular dependency risk and doesn't guarantee seed completion. Better approach: have the beat_bootstrap.py itself handle the case where system_config rows don't exist yet (use fallback defaults, same as the existing fallback in the function). The seed will eventually run from the backend container, and on the next Beat restart the bootstrap will pick up the seeded values. For the initial deployment, the bootstrap's fallback defaults match the seed defaults, so behavior is correct either way.

Final decision for depends_on: Add `postgres: condition: service_healthy` to beat (it needs DB access). Do NOT add `backend` dependency — the bootstrap has its own fallback logic for missing rows.
