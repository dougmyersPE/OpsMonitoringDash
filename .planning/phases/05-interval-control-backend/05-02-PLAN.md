**1. Extract a reusable `update_redbeat_entry()` helper in beat_bootstrap.py:**

Add a standalone function to `beat_bootstrap.py` that can be called from the API process to update a single RedBeat entry in Redis after a DB commit. This function is separate from `bootstrap_beat_schedule()` (which writes all 6 entries).

```python
def update_redbeat_entry(worker_key: str, new_seconds: float) -> None:
    """Update a single RedBeat entry's interval. Called from API after DB commit.

    Uses from_key() to preserve last_run_at, then saves with new schedule.
    Falls back to creating a new entry if one doesn't exist yet.
    """
```

The function:
1. Looks up `beat_name` and `task_name` from `BEAT_NAME_MAP` and `WORKER_TASK_MAP` using `worker_key`
2. Generates the Redis key via `RedBeatSchedulerEntry.generate_key(celery_app, beat_name)`
3. Tries `RedBeatSchedulerEntry.from_key(redis_key, app=celery_app)` to load existing entry (preserves `last_run_at`)
4. Sets `entry.schedule = celery_schedule(new_seconds)`
5. Calls `entry.save()`
6. If `from_key` raises `KeyError`, creates a new `RedBeatSchedulerEntry` and saves it

This function uses sync Redis (RedBeat's internal client). It will be called from an async context via `asyncio.get_event_loop().run_in_executor()` to avoid blocking the event loop (per RESEARCH.md Pitfall 5).

**2. Extend the PATCH handler in config.py:**

Modify `update_config()` to detect interval keys and apply validation BEFORE the DB write. The detection is by prefix: keys matching `poll_interval_*` but NOT ending in `_min` are interval keys.

Add a dict at module level mapping interval key names to their worker key suffix:
```python
# Interval key -> worker key suffix (for looking up _min and updating RedBeat)
INTERVAL_WORKER_KEYS = {
    "poll_interval_prophetx": "prophetx",
    "poll_interval_sports_data": "sports_data",
    "poll_interval_odds_api": "odds_api",
    "poll_interval_sports_api": "sports_api",
    "poll_interval_espn": "espn",
    "poll_interval_critical_check": "critical_check",
}
```

Validation logic (runs BEFORE DB write for interval keys):

a. Parse the new value as an integer. If parsing fails or value <= 0, raise `HTTPException(status_code=422, detail=f"Poll interval must be a positive integer (seconds), got: '{body.value}'")`.

b. Read the minimum from DB: query `SystemConfig` where `key == f"poll_interval_{worker_key}_min"`. If the min row exists, parse its value as int. If `new_seconds < min_seconds`, raise `HTTPException(status_code=422, detail=f"Poll interval for '{key}' must be at least {min_seconds}s (got {new_seconds}s). Minimum enforced to prevent API abuse.")`.

c. After the existing DB commit succeeds, propagate to RedBeat. Import `update_redbeat_entry` from `beat_bootstrap` inside the function body (deferred import to avoid importing celery_app at module level in the API process — per RESEARCH.md Pitfall 3 and STATE.md pending todo). Call it via `asyncio.get_event_loop().run_in_executor(None, update_redbeat_entry, worker_key, float(new_seconds))` and await the result.

d. If the key ends with `_min`, also validate it's a positive integer (same parse check), but do NOT propagate to RedBeat (minimums don't affect Beat entries).

Non-interval keys (e.g., `alert_only_mode`, `slack_webhook_url`) pass through the existing code path unchanged.

IMPORTANT: The `update_redbeat_entry` import must be deferred (inside the function body, not at module top). The STATE.md pending todo says "Confirm that importing celery_app in the FastAPI API process does not trigger worker registration side effects." A deferred import inside the handler function body (not at module level) minimizes this risk — it only runs when an admin actually PATCHes an interval key.

**Note on RedBeat propagation in tests:**
The `update_redbeat_entry` call inside the PATCH handler will try to connect to Redis and write a RedBeat entry. In the test environment, this may fail if Redis is not running or if celery_app is not properly configured. The test should still pass because the RedBeat update happens AFTER the DB commit and response — if it fails, it should be caught and logged, not crash the endpoint.

If RedBeat propagation causes test failures, wrap the `update_redbeat_entry` call in a try/except in config.py that logs the error but does not re-raise (the DB write is the source of truth; RedBeat propagation is best-effort from the API endpoint — the bootstrap will sync on next Beat restart anyway).

**Monkeypatching approach (preferred):** Use `monkeypatch` or `unittest.mock.patch` to mock `update_redbeat_entry` in the test module so tests don't need a live Redis. Mock at the import location in config.py:


Actually, since it's a deferred import inside the handler, mock the source module:
