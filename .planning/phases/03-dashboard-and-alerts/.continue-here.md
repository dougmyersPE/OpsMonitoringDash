## ProphetX WebSocket Consumer (new Docker service)

### What we know from docs:
- Pusher-based WebSocket (not raw WS)
- Config endpoint: GET /partner/websocket/connection-config → returns key, cluster
- Auth endpoint: /partner/mm/pusher (used by pysher for private channel auth)
- Broadcast channel: `private-broadcast-service=3-device_type=5`
- Private channel: `private-service=3-device_type=5-user=<partnerId>`
- Relevant event topic: `sport_event` (bound per tournament: `tournament_{id}`)
- Payload: base64-encoded JSON, wrapper: {timestamp, change_type, payload, op}
- op values: c=create, u=update, d=delete
- Health check pushed every 5 seconds on broadcast channel
- Token TTL: 20 min — must refresh before expiry

### Implementation tasks:
1. Add `pysher` to backend/pyproject.toml dependencies
2. Create `backend/app/workers/ws_prophetx.py`:
   - get_access_token() using ProphetXClient auth flow
   - get_pusher_config(token) → GET /partner/websocket/connection-config
   - handle_sport_event(data) → decode base64 payload, upsert Event in DB, publish SSE
   - run() main loop with reconnect on failure
   - Token refresh logic (re-auth ~60s before 20-min expiry)
3. Add `ws-consumer` service to docker-compose.yml:
   - Same build as backend
   - Command: `python -m app.workers.ws_prophetx`
   - Depends on: postgres (healthy), redis (healthy)
4. Reduce poll_prophetx beat interval from 30s to 5 minutes (reconciliation only)
5. Build and deploy

### Key implementation notes:
- pysher.Pusher() takes: key, cluster, auth_endpoint, auth_endpoint_headers
- Subscribe AFTER pusher:connection_established fires (get socket_id first)
- Channel subscription: pusher.subscribe("private-broadcast-service=3-device_type=5")
- Event binding: channel.bind("sport_event", handle_sport_event)
- sport_event payload.info has same shape as /mm/get_sport_events response:
  {event_id, name, sport_name, status, scheduled, tournament_id, tournament_name,
   competitors[], display_name, type, sub_type, updated_at}
- op=d (delete) → mark event as ended (not in ProphetX anymore)
- Keep the existing stale-event logic in poll_prophetx as-is for reconciliation

<context>
The goal is to replace the 30-second poll_prophetx cycle with real-time push.
ProphetX uses Pusher (a managed WebSocket service). The Python client is `pysher`.

Architecture: new `ws-consumer` Docker service runs forever, connecting to Pusher
and handling sport_event messages by upserting into the DB and publishing SSE updates
(same _publish_update() call as poll_prophetx uses today).

poll_prophetx stays but runs every 5 minutes instead of 30 seconds — it becomes
a reconciliation mechanism to catch any events missed during WebSocket downtime.

The ws_prophetx module should be a standalone script (if __name__ == "__main__")
rather than a Celery task, since it runs as its own process via docker-compose.
</context>
